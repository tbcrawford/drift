<objective>
Fix the silent-acceptance bug in WithContext: negative values are currently
stored unchanged into config.diff.contextLines and silently passed to the hunk
builder, where `lo = idx - contextLines` with a negative contextLines expands
the context window to the entire file.

The fix: validate at the point where contextLines is consumed. The cleanest API-stable
approach for a library is to validate inside Diff() before invoking the hunk builder,
returning an error to the caller. This preserves the `Option func(*config)` signature
(no return value) and follows the same pattern as unknown algorithm values.

Concretely:
1. Add a `validate()` method to `config` that checks `diff.contextLines >= 0`.
2. Call `cfg.validate()` at the top of `drift.Diff()` before any diff work.
3. `drift.Diff()` already returns `(DiffResult, error)` — returning a non-nil
   error on negative context is consistent with the existing API.
4. The CLI's `resolveRootOptions` in `cmd/drift/flags.go` already validates
   `flags.context < 0` and returns an early exit-code error — that path
   remains correct and is now a defense-in-depth check at the CLI layer.
</objective>

```go
// validate checks that all config fields hold valid values.
// It is called by Diff() before any diff work begins.
func (c *config) validate() error {
    if c.diff.contextLines < 0 {
        return fmt.Errorf("drift: WithContext value must be non-negative, got %d", c.diff.contextLines)
    }
    return nil
}
```

Add `"fmt"` to the imports in options.go.

**In drift/drift.go**, add a validate call at the top of Diff() (after config
is assembled, before the diff algorithm is invoked):

```go
if err := cfg.validate(); err != nil {
    return DiffResult{}, err
}
```

Place it immediately after `cfg := applyOptions(opts)` (or wherever the config
is assembled — use the actual variable name found in Task 1).

Do NOT change the WithContext function itself — it continues to store the value.
Validation is deferred to Diff() call time, not option construction time. This
is the standard Go functional-options pattern (validate on use, not on set).
